# Personal Learning Coach 项目实施计划

版本：v1.0
目标：将“私人学习教练”能力建设为一个可独立开发、测试、部署、长期运行的项目
实施方式：分阶段、小步快跑、以可运行闭环优先

## 1. 实施目标

本实施计划的核心目标不是一次性做完所有功能，而是在尽量短的周期内，优先做出一个稳定的最小闭环版本，并在此基础上逐步补齐复习、报告、在线资源、多领域与 UI 能力。

第一优先级闭环：
1. 新领域开始学习
2. 水平测试
3. 生成计划
4. 每日自动推送
5. 用户提交
6. 自动评估
7. 连续推进学习进度
8. 数据持久化

---

## 2. 项目实施原则

### 2.1 闭环优先
优先实现“端到端能跑通”的教学闭环，而不是先做很多边角功能。

### 2.2 规则先行，模型增强
关键业务状态推进由规则保证稳定，生成式模型作为增强层，而不是单点依赖。

### 2.3 数据模型先定，再写功能
因为该项目的核心价值之一是长期持续学习与可追溯评估，数据模型必须先稳定。

### 2.4 先本地单用户，再扩展
先面向单用户、单平台、单领域实现，再扩展多用户、多平台、多领域。

### 2.5 每阶段都有可验证产物
每一阶段结束必须有可演示结果、测试用例和验收清单。

## 5. 详细实施阶段

## 阶段 0：项目初始化（预计 1-2 天）

### 目标
建立独立仓库与基础工程骨架。

### 任务
1. 创建项目仓库
2. 初始化 pyproject.toml
3. 建立 src/tests/docs 目录
4. 配置 lint/format/test 工具
5. 建立基础 README
6. 定义环境变量与配置文件规范

### 交付物
- 可安装项目骨架
- 基础 CI（至少运行测试与 lint）

### 验收标准
- 能执行 pytest
- 能导入主包
- 文档目录可用

---

## 阶段 1：数据模型与存储层（预计 2-4 天）

### 目标
先把最关键的持久化地基打稳。

### 子任务
1. 定义 Pydantic/dataclass 模型
2. 设计用户目录结构
3. 实现 JSON 读写封装
4. 实现原子写入与备份策略
5. 增加 schema_version
6. 实现旧数据自动补齐
7. 明确 assessment_history、submission_history 等必保字段

### 重点文件
- src/personal_learning_coach/models.py
- src/personal_learning_coach/data_store.py
- src/personal_learning_coach/migrations.py

### 测试
1. 新用户初始化测试
2. 多领域目录创建测试
3. 缺失字段补齐测试
4. 损坏文件恢复/报错测试

### 验收标准
- 任意用户/领域可建立独立数据目录
- 历史 JSON 读取时可自动补齐缺失字段
- 原始回答和结构化评分字段可稳定写入

---

## 阶段 2：学习开始与水平测试（预计 2-4 天）

### 目标
实现新领域学习的冷启动能力。

### 子任务
1. 定义 ai_agent 领域测试题模板
2. 实现 start_learning(domain) 流程
3. 实现首轮引导提问流程（包括学习时间、学习风格、推送时间、目标水平等）
4. 实现 assessment start/submit 流程
5. 实现初始水平判定
6. 实现学习偏好参数收集与默认值
7. 把评估结果写入 assessment_history

### 重点文件
- level_tester.py
- coach.py
- prompts/assessment/*.md 或 *.json

### 测试
1. 新用户启动 ai_agent 学习测试
2. 提交测试答案并得到初始等级
3. 无交互场景走默认配置

### 验收标准
- 新用户可以开始一个领域
- 系统能输出初始水平与建议
- 测试原始答案、反馈、结构化评分成功落盘

---

## 阶段 3：学习计划生成（预计 2-5 天）

### 目标
根据水平与偏好生成结构化计划。

### 子任务
1. 定义 ai_agent 课程地图
2. 将课程拆成模块与话题
3. 为每个话题定义目标、概念、练习、问题
4. 基于水平裁剪学习顺序和难度
5. 输出 plan.json 与 progress.json

### 重点文件
- plan_generator.py
- domain_catalog/ai_agent.json 或 python module

### 测试
1. beginner / intermediate / advanced 三档计划生成测试
2. 每日时长不同导致节奏不同测试
3. 理论/实践/结合风格差异测试

### 验收标准
- 可为 ai_agent 生成完整计划
- 计划结构稳定、可续接
- 进度指针可正确指向下一学习内容

---

## 阶段 4：每日学习推送与调度（预计 3-5 天）

### 目标
实现每日学习内容的定时推送与立即推送能力。

### 子任务
1. 实现 content_pusher
2. 实现“选择今日应推内容”的算法
3. 实现推送内容模板渲染
4. 实现 push record 持久化
5. 对接调度器
6. 对接 Telegram/local delivery
7. 支持 run now / pause / resume

### 重点文件
- content_pusher.py
- scheduler_adapter.py
- delivery/telegram.py
- delivery/local.py

### 测试
1. 首次推送第 1 天学习内容
2. 已有进度时推送下一话题
3. 暂停后不再触发
4. 立即执行成功
5. 投递失败有错误记录

### 验收标准
- 每天固定时间自动推送
- 投递结果可查询
- 不会因为资源推荐失败导致推送失败

---

## 阶段 5：提交与评估（预计 3-6 天）

### 目标
把用户回应纳入正式闭环。

### 子任务
1. 定义 submission record
2. 接收原始文本/结构化回答
3. 实现 evaluator 输入封装
4. 实现结构化评分输出
5. 计算 total_score 与 mastery score
6. 写入 evaluations 与 submission_history
7. 生成反馈文本
8. 更新 topic/domain 进度

### 重点文件
- evaluator.py
- mastery_engine.py
- submission handler（可在 coach.py 或独立模块）

### 测试
1. 高质量答案应推进到下一话题
2. 低质量答案应停留并建议复习
3. 漏答关键概念应在 missed_concepts 体现
4. 原始答案与反馈要可追溯

### 验收标准
- 用户提交后能获得评分与反馈
- 系统能自动做推进/复习决策
- 所有关键评估字段均已持久化

---

## 阶段 6：复习与回顾机制（预计 2-4 天）

### 目标
建立真正长期学习所需的回顾机制。

### 子任务
1. 定义 review rules
2. 设计复习任务类型
3. 在每日调度中插入 review due 任务
4. 错题/弱点优先复习
5. 更新 next_review_at 字段

### 测试
1. 低分话题自动进入复习队列
2. 到达时间窗后自动触发复习推送
3. 复习完成后可回到主线学习

### 验收标准
- 至少具备 1/3/7 天基础复习机制
- 复习逻辑不打乱整体进度

---

## 阶段 7：总结与报告（预计 3-5 天）

### 目标
让学习结果可视化、可复盘。

### 子任务
1. 设计报告 JSON 汇总层
2. 生成 HTML 报告
3. 输出完成度、趋势图、强弱项、常错点
4. 支持周报/阶段报告/结业报告
5. 生成总结文本

### 重点文件
- report_generator.py
- templates/report.html

### 测试
1. 无学习记录时报告可优雅处理
2. 多次提交后趋势图数据正确
3. 报告文件生成路径正确

### 验收标准
- 可以生成可打开的 HTML 报告
- 报告能准确反映计划、进度与评分趋势

---

## 阶段 8：综合测试与结业判断（预计 2-4 天）

### 目标
实现“学会了没有”的最终判断机制。

### 子任务
1. 定义 final assessment 触发规则
2. 设计综测题
3. 执行综测评估
4. 根据结果决定完成/返修
5. 输出结业总结

### 验收标准
- 达标用户可完成领域学习
- 未达标用户得到明确补学建议

---

## 阶段 9：在线资源增强（预计 2-4 天）

### 目标
让每日内容具备时效性。

### 子任务
1. 封装 online resource fetcher
2. 增加缓存层
3. 资源分类与去重
4. 抓取失败降级
5. 推送中插入推荐资源块

### 验收标准
- 有网时可推荐资源
- 无网/失败时主链路不受影响

---

## 阶段 10：API 与集成接口（预计 3-6 天）

### 目标
便于项目独立部署和外部调用。

### 子任务
1. 使用 FastAPI 暴露核心接口
2. 定义请求/响应 schema
3. 接入认证（可选首版简化）
4. 增加健康检查与状态接口

### 验收标准
- 可通过 API 完成 start/push/submit/report 基础流程

---

## 6. 测试策略

## 6.1 单元测试
覆盖：
- 数据模型
- 迁移逻辑
- 评分计算
- 状态推进
- 复习规则

## 6.2 集成测试
覆盖：
1. 新用户 -> 测试 -> 计划 -> 推送
2. 推送 -> 提交 -> 评估 -> 更新进度
3. 低分 -> 复习 -> 再评估
4. 高完成度 -> 综测 -> 结业

## 6.3 回归测试
重点防止：
- 持久化字段丢失
- 历史数据无法读取
- 定时任务重复推送
- deliver target 配置错误

## 6.4 手工验收测试
建议准备固定脚本场景：
- beginner 用户
- intermediate 用户
- 多领域用户
- 推送失败用户
- 中断恢复用户

## 7. 实施顺序建议（最优先路线）

如果资源有限，按以下顺序开发：
1. 数据层
2. 水平测试
3. 计划生成
4. 每日推送
5. 提交评估
6. 进度推进
7. 报告
8. 复习
9. 在线资源
10. API/UI

原因：前 6 项才构成核心商业价值闭环。

## 9. 风险控制计划

### 9.1 数据丢失风险
应对：
- 原子写入
- 版本字段
- 定期备份
- 写前校验

### 9.2 LLM 输出不稳定
应对：
- 强约束 JSON 输出
- rubric 固化
- fallback 规则评分

### 9.3 调度/投递失败
应对：
- 任务状态跟踪
- 失败重试
- 幂等控制
- 可人工立即重跑

### 9.4 范围膨胀
应对：
- v1 只做单用户 + ai_agent + Telegram + HTML 报告
- 其他全部放入后续版本

## 15. 项目成功标准

项目达到成功状态的判断标准：
1. 用户无需每天主动提醒，系统会定时推送学习任务
2. 用户的每次回答与实践结果都能被保存、评估、追踪
3. 学习路径不会中断，能持续接着上次进度推进
4. 系统能知道什么时候该继续、什么时候该复习、什么时候该测试
5. 用户能随时看到学习成果、薄弱点和下一步建议
